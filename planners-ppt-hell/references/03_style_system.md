# Visual Design System

本文档只用于 DRAFT/SVG 阶段。它负责让已批准的 `layout_plan.json` 变得清楚、成熟、统一、有记忆点。

一句话职责：

> 版式已定，视觉深化。

本文档不重新选择版式，不重新决定上屏文案，不改变 wireframe 主结构。如果视觉执行发现版式无法成立，应回到 PLAN，而不是在 SVG 中私自改结构。

SVG/PPT 兼容、禁用元素、metadata、文本属性和 validator 工程红线看 `04_svg_rules.md`。

---

## 1. DRAFT 必须回答的 5 个问题

每页生成 SVG 前，必须回答：

### 1. 已批准的信息锚点是什么？

信息锚点必须来自 `layout_plan.json`，可能是：

- 标题判断。
- 关键数字。
- 截图、图表或案例证据。
- 对比关系。
- 流程节点。
- 结论句。
- 模型结构。

每页都需要一个明确视觉重点，但它必须来自已批准的内容/结构判断。

### 2. 阅读路径如何被视觉强化？

使用以下手段强化已批准的 reading path：

- 字号层级。
- 位置与面积。
- 留白。
- 色彩语义。
- 线条、轴线或编号。
- 图片、截图或图表。
- 结论装置。

视觉不能让读者绕开真正的信息锚点。

### 3. 组件语言是否服务内容关系？

先判断内容关系，再选组件：

- 开放式排版。
- 轻线分割。
- 编号系统。
- 轴线 / 时间线。
- 证据板。
- 图表 + 解读。
- 卡片容器。

不要默认把所有内容装进圆角卡片。

### 4. 这一页和整份 deck 是否统一但不机械重复？

统一来自：

- 稳定画布。
- 稳定边距。
- 稳定字体体系。
- 稳定主色和辅助色。
- 稳定 source / 页码位置。
- 稳定标题入口。

变化来自：

- 信息重心。
- 模块节奏。
- 证据呈现。
- 页面呼吸。
- 情绪节点。

避免每页同款白底圆角卡片、同款底部结论条、同款装饰抢注意力。

### 5. 是否为了好看牺牲了可读性、证据或可编辑性？

如果视觉效果导致以下问题，设计失败：

- 证据不可读。
- 正文过小。
- 颜色没有语义。
- 装饰压过内容。
- Source 侵入正文。
- 图片只是填空。
- 视觉执行静默改变了已批准版式。

## 2. 不可让步的视觉底线

- 服从已批准的 `layout_plan.json`。
- 使用 `copy_handling.final_on_slide` 作为唯一上屏文案来源。
- 标题、证据、解释、结论必须有清楚层级。
- 每页必须有一个视觉重点。
- 重复组件必须属于同一套系统。
- 可以有设计张力，但不能牺牲可读性、可编辑性和 SVG/PPT 兼容性。
- 如果内容放不下，回到 PLAN，不靠缩字、裁切或静默删文案解决。

## 3. 硬性视觉规范

这些规范用于守住专业 PPT 的最低质量线。它们不是版式模板，但默认应执行，除非用户模板或品牌规范明确覆盖。

### 3.1 标题入口装置

理性页默认使用标题左侧深红竖线作为 deck 身份锚点。

规范：

- 位置：主标题左侧，与标题第一行顶部对齐。
- 宽度：`4px`。
- 高度：`48-76px`，约等于标题块高度的 `70%`。
- 颜色：`#C62828`。
- 与标题间距：`18-24px`。
- 适用：`rational` 页面默认使用。
- 可省略：封面、章节、引语、强情绪页、全图页，或模板已有明确标题装置时。

限制：

- 不要扩展成大色块或左侧栏。
- 不要在每个模块标题旁重复小红条。
- 不要让标题装置抢走内容锚点。

### 3.2 字体与字号

正文默认字体：

```text
Microsoft YaHei, SimHei, sans-serif
```

标题可根据页面模式选择：

- `rational`：无衬线，清晰、克制、工程化。
- `emotional`：可用 `Georgia` 或 `SimSun` 做编辑感和停顿。

推荐字号：

| 角色 | SVG font-size | 说明 |
| --- | --- | --- |
| 主行动标题 | `48-56px` | 必须明显高于正文，形成主要入口 |
| 副标题 / 导语 | `26-32px` | 支撑标题，不抢主标题 |
| 模块标题 | `24-30px` | 分组或卡片标题 |
| 正文 | `22-26px` | 默认阅读字号 |
| 紧凑正文 | `20-22px` | 正文下限，只用于空间紧张处 |
| 注释 / 来源 | `18-20px` | 低调但可读 |
| KPI / 大数字 | `64-120px` | 必须成为主要视觉锚点 |
| 情绪化大句 | `72-128px` | 感性页专属，字数必须少 |

底线：

- 正文不得低于 `20px`。
- 任何有意义文字不得低于 `16px`。
- 单页不超过 4 个主要字号档位。
- 正文行高约为 `1.5-1.6` 倍字号。
- 放不下时回到 PLAN，不靠缩字硬塞。

### 3.3 色彩克制

默认色彩：

| Token | Hex | 用途 |
| --- | --- | --- |
| 背景 | `#F5F7FA` / `#FFFFFF` | 默认页面底色 |
| 深色背景 | `#051C2C` | 章节、封面、情绪页底色 |
| 主标题 | `#333333` | 标题、大判断 |
| 正文 | `#555555` | 正文、说明 |
| 注释 | `#999999` | source、脚注、低强调信息 |
| 卡片 | `#FFFFFF` | 内容容器 |
| 边线 | `#E0E0E0` / `#CCCCCC` | 分割线、卡片描边 |
| 主强调 | `#C62828` | 关键数据、结论、标题锚点、关键标注 |

硬规则：

- 单页最多一个强强调色。
- `#C62828` 只用于核心数据、结论、标题锚点或关键标注。
- 不使用红黄绿蓝多色 badge 做分类。
- 多类别区分优先用灰度、线条、位置、编号，而不是多色块。
- 深色页必须保证文字对比度。
- 颜色必须有语义：强调、分组、风险、结论或证据。

### 3.4 Source / 页脚

Source、脚注和页码属于页面外壳，不属于正文内容区。

规范：

- 默认放左下或右下安全区。
- `y` 通常在 `1010-1030`。
- 字号 `18-20px`。
- 颜色 `#999999`。
- Source 只保留必要来源、日期、口径或注释，不写成长段正文。
- 不侵入底部结论条、图表坐标轴、图片说明区或正文区。

如果某页无来源，也应保持页脚留白秩序，避免其他内容压到页面底部。

### 3.5 组件纪律

卡片不是默认答案。

选择优先级：

- 能用留白和对齐解决的，不加框。
- 需要分组时，优先轻线分割。
- 需要比较、承托截图或同级模块扫描时，再使用卡片。
- 顺序关系优先编号、轴线、阶梯或流程线。
- 证据关系优先证据板、截图阵列或图表 + 解读。

卡片规范：

- 同级卡片尺寸一致。
- 内边距一致。
- 标题位置一致。
- 描边克制：`#E0E0E0` / `#CCCCCC`，`1px`。
- 理性页圆角通常 `6-12px`。
- 图片/截图卡片可用 `12-16px`。
- 不连续多页机械使用同款白底圆角卡片。

### 3.6 情绪页限制

`emotional` 页面限制：

- 只允许一个主视觉焦点。
- 主标题建议少于 25 字。
- 副标题建议少于 40 字。
- 保留 `20%-30%` 连续留白。
- 不承载复杂证据链。
- 不牺牲可读性换取设计感。
- 如果正文超过约 100 字，通常应降级为理性页设计。

## 4. 组件语言选择

组件语言应来自已批准的内容关系。

### 开放式排版

适合：单一强判断、结论页、情绪页、引语页。

依靠字号、留白、对齐和少量强调建立层级，不用容器填满画面。

### 轻线分割

适合：理性分析页、方法页、列表页、轻量并列信息。

用 `1px` 浅灰线建立分组，保持页面安静。

### 编号系统

适合：步骤、优先级、建议、方法拆解。

编号应帮助读者理解顺序或层级，不只是装饰。

### 轴线 / 时间线

适合：流程、阶段、前后演进、机制。

轴线必须表达真实顺序、时间或因果，不要把无序 bullet 强行串起来。

### 证据板

适合：竞品截图、用户研究、案例证明、素材佐证。

截图比例要统一，图注要解释证据，不要只贴标签。

### 图表 + 解读

适合：数据形态和洞察同样重要。

图表要可读，解读要说出洞察，不要重复图表标题。

### 卡片容器

适合：同级信息比较、截图承托、多模块扫描。

卡片必须表达边界或关系，不能只是为了显得有设计。

## 5. 按页面模式执行

### `rational`

视觉语气：

- 清楚。
- 稳定。
- 有秩序。
- 可扫描。
- 证据优先。

默认做法：

- 使用标题锚点竖线。
- 使用克制商务色。
- 使用明确网格、轴线或分割关系。
- 保证正文、图表、截图可读。

### `emotional`

视觉语气：

- 有停顿。
- 有张力。
- 有记忆点。
- 单一主角。
- 留白明显。

默认做法：

- 可以省略标题锚点竖线。
- 可以使用深色背景或更大字号。
- 可以使用衬线标题。
- 保持文案短，避免复杂论证。

## 6. PNG 视觉自检

看实际 PNG，而不是想象中的 SVG。

每页自检时问：

- 这一页是在开场、诊断、举证、比较、转折，还是收束？
- 用户看完这一页，应该改变什么判断？
- 第一眼应该看到什么：标题、数字、图表、对比关系，还是一句判断？
- 信息锚点是否足够突出？
- 阅读路径是否自然？
- 标题、证据、解释、结论是否分层清楚？
- 组件是否表达了内容关系？
- 留白是节奏，还是空洞？
- 颜色是否有语义？
- 这页是否又落回了浅灰底、白卡片、红色编号、底部结论条的安全模板？
- 如果不用底部结论条，这页是否仍然成立？只有当结论条承载明确观点时才使用它。
- Source 是否低调且不侵入正文？
- 这页是否忠实于 `layout_plan.json`？
- 这页是否和整份 deck 属于同一套系统？

如果视觉问题需要改变主结构、拆页或改文案，回到 PLAN。不要在 SVG 阶段静默重排或删除内容。

---

## 7. 模板或品牌规范覆盖

如果用户提供 PPT 模板或品牌规范：

- 以模板色彩、字体、页眉页脚、logo 安全区为准。
- 保留本文档的视觉底线：可读性、层级、统一、色彩克制、内容优先。
- 不要把模板当成版式锁；内容区仍需服从已批准的 `layout_plan.json`。
- 如果模板和 SVG/PPT 兼容规则冲突，以 `04_svg_rules.md` 为准，并记录取舍。
