# Layout Thinking

本文档只用于 PLAN 阶段。它负责让模型判断：一页内容应该如何被组织。

一句话职责：

> 先证明你理解了这一页的信息关系，再选择版式。

本文档不决定字体、颜色、卡片样式、阴影、装饰、SVG 写法或最终视觉风格。这些属于 DRAFT/SVG 阶段的 `03_style_system.md` 和 `04_svg_rules.md`。

`layout_id` 记录的是主导组织逻辑，不是模板坐标。PLAN 阶段的输出是已批准的 `layout_plan.json`；进入 DRAFT/SVG 后不得重新打开本文档挑选版式家族。

---

## 1. PLAN 必须回答的 6 个问题

每一页进入 `layout_plan.json` 前，必须回答：

### 1. 这页的说服动作是什么？

可选动作：

- `define`：定义问题、对象、边界或概念。
- `prove`：用证据证明一个判断。
- `compare`：解释对象、状态、方案之间的差异。
- `sequence`：表达时间、步骤、流程或阶段。
- `structure`：建立模型、层级或系统。
- `summarize`：把复杂信息压缩成决策、建议或结论。
- `emotionalize`：制造停顿、情绪、转折或记忆点。

### 2. 内容之间是什么关系？

可选关系：

- `single_claim`：一个强判断需要空间和强调。
- `parallel_set`：多个项目地位相等。
- `weighted_set`：多个项目相关但不等权。
- `compare`：两个或多个对象有可比维度。
- `sequence`：时间、流程、漏斗、阶段很重要。
- `hierarchy`：层级、优先级、依赖或嵌套逻辑很重要。
- `evidence_chain`：判断需要数据、案例、截图或图表证明。
- `matrix`：两条轴线或分类解释内容。
- `summary`：复杂材料需要收束成行动或判断。

同一份内容可能支持多种关系。最终选择必须在 `layout_reason` 中说清楚。

### 3. 谁是信息主角？

必须明确本页的信息锚点：

- 一个结论？
- 一个数字？
- 一张截图？
- 一个对比关系？
- 一个流程节点？
- 一个模型结构？
- 还是多个同级要点？

如果存在明显主角，版式应给它更大面积、更早阅读入口或更高视觉权重。如果所有内容等权，也必须说明为什么等权成立。

### 4. 读者的阅读路径是什么？

每页必须能说明：

1. 第一眼看什么？
2. 第二步读什么证据、结构或解释？
3. 最后应该记住什么？

如果阅读路径只能说成“所有内容并列看”，通常说明空间层级还没想清楚。

### 5. 为什么这个版式比另外两个诱惑选项更对？

不允许只写“清晰”“美观”“适合展示”。

必须说明为什么没有选择常见替代方案，例如：

- 为什么不是三栏卡片？
- 为什么不是左右分栏？
- 为什么不是纯结论页？
- 为什么不是流程图？
- 为什么不是证据板？
- 为什么不是图表 + 解读？

这个判断应进入 `why_this_layout`、`why_not_other_layouts`、`adaptation_note` 或 `anti_laziness_check`。

### 6. 哪些内容上屏，哪些进入 notes，为什么？

PLAN 阶段必须把源文案处理成明确的 `copy_handling.final_on_slide`。

原则：

- 保留主张、关键证据、数字、来源、限定条件、风险和业务判断。
- 可以精简重复铺垫、口语化解释、过长修饰语和次要背景。
- 可以把长段落拆成短句、标签、证据点或结论句。
- 可以把完整解释移入 `speaker_notes`，但必须在 `moved_to_notes` 中说明。
- 不得为了视觉好看删掉反例、风险、条件或不确定性。
- 如果精简后只剩口号而证据链消失，应调整版式或拆页，而不是继续压缩。

用户批准版式后，DRAFT/SVG 阶段不得静默改写 `final_on_slide`。

## 2. 页面模式

### `rational`

用于数据、分析、方法、策略、比较、建议、执行路径。

要求：

- 证据可读。
- 层级清楚。
- 信息可以快速扫描。
- 保留事实、数字、来源和限定条件。
- 不追求夸张视觉，不用复杂几何穿插替代清晰表达。
- 不把复杂内容伪装成情绪页。

### `emotional`

用于封面、章节、转场、引语、宣言、强问题页、结束页。

要求：

- 字数少。
- 单一记忆点。
- 有停顿、张力或叙事转折。
- 不承载复杂论证。
- 不牺牲可读性换取设计感。
- 在整份 deck 中只用于关键节点，通常不应占比过高。

严禁错位：不要为了好看把复杂理性页做成情绪页，也不要为了省事把需要停顿的转折页做成平庸列举页。

## 3. 反偷懒判断

模型最常见的 layout 偷懒有 6 种：

1. 把任何内容都做成三栏卡片。
2. 把两段文字放成左右两栏，却没有真实比较关系。
3. 有关键证据，却不给证据足够面积。
4. 有明显主角，却把所有内容做成等权。
5. 连续多页重复同一骨架，却没有节奏理由。
6. 文案放不下时，假装 SVG 阶段可以解决。

每页必须在 `anti_laziness_check` 或 `layout_reason` 中证明自己没有犯这些问题。

节奏提示：

- 如果连续三页 `page_mode + visual_density` 相同，需要解释原因或调整节奏。
- 如果多页连续使用卡片网格，只要内容允许，应引入不同组织方式。
- 长 deck 适合章节页；短 proposal 不应为了完整感硬插转场。

## 4. 版式家族索引

以下是组织逻辑索引，不是硬模板。选择 `layout_id` 后，必须说明本页如何适配真实文案、证据、容量和整份 deck 节奏。

### L01 - Cover / Opening Claim

用于：开启整份 deck，建立核心主题或主判断。

使用前问：

- 观众必须记住的一句话是什么？
- 辅助信息是否足够短？
- 是否需要素材区，还是只需要文字张力？

偷懒信号：封面塞进目录级信息；背景装饰和主题无关。

### L02 - Section / Transition

用于：换章、停顿、叙事转向或承诺下一段内容。

使用前问：

- 上一章节和下一章节之间发生了什么变化？
- 这是停顿、转折，还是承诺？
- 章节标题能否更短、更有记忆点？

偷懒信号：重复上一页标题；正文过多。

### L03 - KPI / Big Number

用于：少数关键数字就是主要证据。

使用前问：

- 哪个数字是主角？
- 指标之间是同级、顺序，还是共同证明一个结论？
- 是否应该用图表，而不是指标块？

偷懒信号：所有数字一样大；没有解释数字证明了什么。

### L04 - Claim + Evidence

用于：一个主判断需要截图、图表、案例、图片或数据支撑。

使用前问：

- 主判断是什么？
- 哪个证据让它可信？
- 论点和证据谁应获得更大面积？
- 证据是真实素材、占位素材，还是生成素材？

偷懒信号：旁侧视觉只是装饰；正文过长导致证据变小。

### L05 - Evidence Grid

用于：多个视觉、案例或截图共同证明一个模式。

使用前问：

- 这些证据是否同级？
- 观众应该从它们之间看见什么模式？
- 是否需要一个主证据更大？
- 图注是在解释证据，还是只贴标签？

偷懒信号：截图堆砌无解释；比例混乱像随手粘贴。

### L06 - Process / Mechanism

用于：步骤、交接、阶段、运行机制或时间顺序。

使用前问：

- 顺序是时间、运营动作，还是因果？
- 每一步发生了什么变化？
- 结果是否可见？
- 一页是否承载了太多步骤？

偷懒信号：把无序 bullet 强行塞进箭头；每一步都是空泛动词。

### L07 - Problem / Tension

用于：提出战略问题、矛盾点、悬念或转折。

使用前问：

- 希望观众感到什么张力？
- 问题是否具体，且和项目直接相关？
- 下一页是否回答这个问题？

偷懒信号：泛泛的修辞问题；视觉戏剧性没有叙事作用。

### L08 - Quote / Insight

用于：一句话本身就是页面主体。

使用前问：

- 这句话是否强到足以撑起一页？
- 是否需要署名或上下文？
- 语气应该安静、戏剧化，还是编辑感？

偷懒信号：普通总结被放大成金句页。

### L09 - Compare / Before-After

用于：两个对象、状态、策略、竞品或能力存在有意义可比性。

使用前问：

- 两侧是否等权，还是一侧应被强调？
- 比较维度是否对称？
- 这是旧/新、竞品/品牌、问题/解法，还是互补能力？
- 两侧是否都需要证据？

偷懒信号：把两段话放成两列但没有真正比较；重要性不同却默认等宽。

### L10 - Deep Explanation + Support

用于：较复杂解释需要一个辅助视觉、模型、示意图或案例支撑。

使用前问：

- 哪些内容可以移入 notes？
- 解释中哪一部分最需要视觉支撑？
- 支撑视觉是证据、机制，还是记忆装置？

偷懒信号：用小字号代替文案处理。

### L11 - Conclusion / Recommendation

用于：把前面证据收束为决策、建议、行动或下一步。

使用前问：

- 决策含义是什么？
- 建议之间是并列、顺序，还是有优先级？
- 需要结论条，还是整页本身就是结论？

偷懒信号：重复上一页；summary 变成密集复述。

### L12 - Three Pillars

用于：三个同级概念确实构成框架。

使用前问：

- 它们真的是同级支柱吗？
- 统一它们的上位逻辑是什么？
- 是否有一个需要更强强调？
- 三卡片是否对这个 deck 时刻太可预测？

偷懒信号：任意三条 bullet 被包装成三大支柱。

### L13 - Hierarchy / System Layer

用于：内容有层级、依赖、优先级或嵌套结构。

使用前问：

- 哪一层支撑哪一层？
- 层级是自上而下、自下而上，还是中心向外？
- 读者应该从哪里进入这个系统？

偷懒信号：只是为了让列表看起来更有结构而堆起来。

### L14 - Chart + Interpretation

用于：数据形态和解释同样重要。

使用前问：

- 数据洞察是什么，而不是图表类型是什么？
- 图表应占主导，还是与解读平衡？
- 哪些数据标签必须上屏？
- 哪些内容应进入 notes？

偷懒信号：图表贴上去没有解释；解读只是重复标签。

### L15 - Closing / Final Action

用于：用最终记忆点、行动或联系方式收束 deck。

使用前问：

- deck 结束后应该留下什么？
- 这是感谢页、宣言、决策页，还是下一步？
- 需要联系方式，还是只需要最终判断？

偷懒信号：泛泛的 Thank you，没有项目特异性。

### L00 - Custom

用于：L01-L15 都不能准确描述本页组织逻辑。

使用前问：

- 为什么参考家族不足以表达这页关系？
- 自定义结构的阅读路径是什么？
- 它是否仍能被 wireframe 和 copy_handling 清楚记录？

偷懒信号：用 custom 回避说明版式选择。

## 5. `layout_plan.json` 中的判断表达

版式选择应体现为“参考家族 + 本页适配”，而不是只填一个编号。

建议每页包含这些判断：

```json
{
  "layout_id": "L09",
  "layout_usage": "adapted",
  "design_judgment": {
    "persuasion_action": "compare",
    "content_relation": "dual-capability contrast",
    "information_anchor": "two named capabilities",
    "reader_takeaway": "品牌不应复制竞品规模，而应补上两种关键能力。"
  },
  "why_this_layout": "本页比较两种战略能力，因此参考 L09 的对比组织逻辑。",
  "why_not_other_layouts": "三支柱会误导为三个同级概念；流程图会虚构顺序。",
  "adaptation_note": "两侧是互补能力而非旧/新关系，因此内容区采用双论证分栏。",
  "anti_laziness_check": "这不是两个泛泛卡片；每侧都有定义、行动含义和证据。"
}
```

`layout_reason` 建议包含：

```text
PageMode: [rational/emotional]
Purpose: [define/prove/compare/sequence/structure/summarize/emotionalize]
InformationAnchor: [主记忆点或主信息锚点]
ReadingPath: [第一眼 -> 第二步 -> 最后记住]
SpatialPlan: [主要区域如何分布，为什么这样分配面积]
Reason: 因为本页内容关系是 [...]，所以采用 [...] 组织方式；上屏内容与 notes 的取舍是 [...]。
```

这不是固定句式，而是防止模型只写空泛理由。

## 6. 失败退回规则

如果 PLAN 阶段发现：

- 内容放不下。
- 证据没有空间。
- 页面关系说不清。
- 上屏文案失真。
- 阅读路径不能明确说明。
- 只能靠小字号硬塞。
- 需要改变主版式、区域比例、拆页或上屏取舍。

不要进入 SVG。应调整 `copy_handling`、wireframe、拆页或移入 speaker notes，并重新生成 `layout_plan.json` 和容量预检。

如果 DRAFT/SVG 阶段发现上述问题，也应记录原因并回到 PLAN/用户确认，而不是在 SVG 中静默改结构。
